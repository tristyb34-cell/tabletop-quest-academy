## Why C++ Plugins at All?

Blueprints are fantastic. You built your first Tabletop Quest prototypes entirely in Blueprints, and they work. So why bring C++ into the picture?

Three reasons.

### Performance Where It Counts

Blueprint nodes are interpreted at runtime. C++ compiles to native machine code. For most of your game, the difference does not matter. A button click, a menu transition, a camera movement: Blueprints handle these just fine. But for systems that run every single frame, or process hundreds of entities at once, C++ can be 10x to 100x faster.

Think about what your combat system will eventually do. Dozens of enemies, each with status effects ticking every frame, each checking line of sight, each evaluating their AI behavior tree. That is where C++ earns its keep. The per-frame cost of Blueprint interpretation adds up when you multiply it across 40 combatants running 10 checks each.

### Engine Features You Cannot Reach from Blueprints

Some Unreal Engine features are only accessible from C++. Custom animation nodes, low-level networking hooks, certain rendering pipeline extensions, and advanced gameplay framework overrides all require C++ code. When you need these features for Tabletop Quest (and you will, especially for the tabletop zoom transition and dual combat mode switching), Claude writes the C++ and you expose it to Blueprints.

### Clean Architecture

As Tabletop Quest grows, you will have dozens of systems: combat, inventory, AI DM communication, tabletop zoom transitions, save/load, quest tracking. Defining the core logic in C++ and exposing clean Blueprint interfaces keeps your project organized. The C++ handles the "how" while Blueprints handle the "what" and "when." It is the difference between a workshop with labelled drawers and one with everything dumped on the floor.

### The Claude-Powered Workflow

Here is the pattern you will use throughout development:

1. You describe what you need in plain English
2. Claude generates the C++ code with proper UE5 macros
3. You paste the code into your project
4. You compile (one button press)
5. You wire it up in Blueprints

That is it. No memorizing C++ syntax. No fighting with template metaprogramming. No late nights debugging pointer arithmetic. Claude handles the C++ complexity, and you focus on game design.

## Understanding UE5 Plugins: The Big Picture

Before we create our first plugin, let's understand how plugins work in Unreal Engine 5.7.

### What is a Plugin?

A plugin is a self-contained bundle of code (and optionally content like meshes, textures, or Blueprints) that extends the engine or your game. Plugins live in a `Plugins/` folder inside your project directory, and each one has its own source files, assets, and configuration.

Think of plugins like expansion packs for your project. Your main game code lives in the `Source/` folder. Plugins sit alongside it, adding capabilities without cluttering your core codebase. You can enable, disable, share, and reuse plugins independently.

### Plugin vs. Module

You will hear both terms used in UE5. Here is the distinction:

- A **module** is a compilation unit. It has a `.Build.cs` file that tells the Unreal Build Tool what source files to compile and what dependencies to link against.
- A **plugin** is a container that holds one or more modules, plus a `.uplugin` descriptor file that the engine reads at startup.

For Tabletop Quest, we will create plugins where each plugin contains a single module. Simple, clean, focused. One plugin per game system.

### Why Plugins Instead of Adding C++ to the Main Project?

You could add C++ classes directly to your game's `Source/` folder. That works for small projects. But plugins give you several advantages:

- **Isolation**: If a plugin's code has a bug, you can disable the entire plugin without affecting the rest of the project. Try doing that with 50 classes scattered through your Source folder.
- **Reusability**: You can copy a plugin folder to another project. Your dice roller works for any RPG, not just Tabletop Quest.
- **Organization**: Each system lives in its own folder with its own dependencies. When you ask Claude to write the inventory system, it does not need to know anything about the combat plugin.
- **Collaboration**: If you ever work with someone else, they can work on a plugin without touching your main code.
- **Claude-friendly**: When you ask Claude for a complete system, Claude can give you a self-contained plugin with everything in one place. No figuring out where to scatter files across your project.

## Setting Up Your Project for C++

If your Tabletop Quest project is currently Blueprint-only, you need to convert it to a C++ project. This is a one-time setup.

### Converting a Blueprint Project to C++

1. Open your Tabletop Quest project in UE5.7
2. Go to **Tools > New C++ Class**
3. Select **Actor** as the parent class (we will delete this actor later; it is just to trigger the conversion)
4. Name it `TempSetupActor`
5. Click **Create Class**

UE5 will:
- Create a `Source/` folder in your project directory
- Generate a `.Build.cs` file for your project module
- Generate `.Target.cs` files for the editor and game
- Open your IDE (Visual Studio, Rider, or Xcode)

You can delete `TempSetupActor.h` and `TempSetupActor.cpp` afterward. The important thing is that the Source folder and build files now exist.

### Your IDE

UE5 supports several IDEs:

- **Visual Studio 2022** (Windows): The standard. Install the "Game development with C++" workload. This is what you should use on your home PC.
- **JetBrains Rider**: Excellent UE5 support, faster than Visual Studio, but costs money.
- **Visual Studio Code**: Works with the right extensions, but less integrated than the above options.
- **Xcode** (Mac): Required on macOS.

For pasting Claude's generated code, your IDE choice does not matter much. You just need it open to create files and hit Build.

Let's build something real: a **TabletopDice** plugin for Tabletop Quest. Every DnD game needs dice, and this plugin will handle all the rolling mechanics your combat system will use in Module 04.

### Step 1: Create the Plugin in UE5

Open your Tabletop Quest project. Go to **Edit > Plugins**. In the Plugins window, click the **+ Add** button in the top-left corner.

Choose **Blank** as the template. Fill in:
- **Name**: `TabletopDice`
- **Author**: Your name
- **Description**: "Dice rolling system for Tabletop Quest"

Click **Create Plugin**. UE5 generates the following file structure inside your project directory:

```
Plugins/

    Source/
      TabletopDice/
        TabletopDice.Build.cs
        Public/
          TabletopDice.h
        Private/
          TabletopDice.cpp
    TabletopDice.uplugin
    Resources/
      Icon128.png
```

### Step 2: Understand the Generated Files

Let's look at each file so you know what they do. You will rarely need to edit most of these yourself. Claude handles them. But understanding the structure helps when something goes wrong.

**TabletopDice.uplugin** is the plugin descriptor. It tells the engine the plugin's name, version, and which modules it contains:

```json
{
    "FileVersion": 3,
    "Version": 1,
    "VersionName": "1.0",
    "FriendlyName": "Tabletop Dice",
    "Description": "Dice rolling system for Tabletop Quest combat and skill checks",
    "Category": "Tabletop Quest",
    "CreatedBy": "Your Name",
    "Modules": [
        {
            "Name": "TabletopDice",
            "Type": "Runtime",
            "LoadingPhase": "Default"
        }
    ]
}
```

**TabletopDice.Build.cs** is the build configuration. This C# file (yes, Unreal's build system uses C# for configuration) tells the Unreal Build Tool what dependencies your plugin needs:

```csharp
using UnrealBuildTool;

public class TabletopDice : ModuleRules
{
    public TabletopDice(ReadOnlyTargetRules Target) : base(Target)
    {
        PCHUsage = PCHUsageMode.UseExplicitOrSharedPCHs;

        PublicDependencyModuleNames.AddRange(new string[]
        {
            "Core",
            "CoreUObject",
            "Engine"
        });
    }
}
```

The `PublicDependencyModuleNames` array lists which engine modules your plugin uses. Almost every plugin needs `Core`, `CoreUObject`, and `Engine`. When Claude generates code that uses additional engine features (like the Gameplay Ability System or Enhanced Input), Claude will tell you to add the relevant module names here.

**TabletopDice.h / TabletopDice.cpp** are the module implementation files. They handle module startup and shutdown. For most plugins, you leave these exactly as generated.

### Step 3: Ask Claude for the Dice Roller Code

Here is exactly how you would prompt Claude to generate the dice roller:

> "Write me a UE5 C++ class for a plugin called TabletopDice. The class should be called UDiceRollerSubsystem and extend UGameInstanceSubsystem. It needs Blueprint-callable functions for:
> - Rolling NdM dice (e.g., 2d6, 1d20) and returning the total plus each individual roll
> - Rolling with advantage (roll twice, take the higher total)
> - Rolling with disadvantage (roll twice, take the lower total)
> - Rolling a stat check against a DC (difficulty class) and returning pass/fail plus the roll value
> - Convenience functions for RollD4, RollD6, RollD8, RollD10, RollD12, and RollD20
> - Getting the last roll result for display in the UI
> - Tracking roll history (capped at 100 entries)
> Use UFUNCTION(BlueprintCallable) macros on everything. Create a USTRUCT for the roll result that includes the total, individual rolls, number of dice, number of sides, and any modifier."

Notice how specific the prompt is. The more detail you give Claude, the better the output. You are not asking "write me a dice roller." You are specifying the exact class hierarchy, the exact functions, the exact data structures, and the exact UE5 patterns to use.

Claude generates something like this:

**DiceRollerSubsystem.h**
```cpp
#pragma once

#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "DiceRollerSubsystem.generated.h"

USTRUCT(BlueprintType)
struct FDiceRollResult
{
    GENERATED_BODY()

    UPROPERTY(BlueprintReadOnly, Category = "Dice")
    int32 Total = 0;

    UPROPERTY(BlueprintReadOnly, Category = "Dice")
    TArray<int32> IndividualRolls;

    UPROPERTY(BlueprintReadOnly, Category = "Dice")
    int32 NumDice = 0;

    UPROPERTY(BlueprintReadOnly, Category = "Dice")
    int32 NumSides = 0;

    UPROPERTY(BlueprintReadOnly, Category = "Dice")
    int32 Modifier = 0;
};

UCLASS()
class TABLETTOPDICE_API UDiceRollerSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    // Roll NdM dice (e.g., Roll(2, 6) = 2d6)
    UFUNCTION(BlueprintCallable, Category = "Dice")
    FDiceRollResult Roll(int32 NumDice, int32 NumSides, int32 Modifier = 0);

    // Roll with advantage: roll twice, take the higher result
    UFUNCTION(BlueprintCallable, Category = "Dice")
    FDiceRollResult RollWithAdvantage(int32 NumDice, int32 NumSides,
                                       int32 Modifier = 0);

    // Roll with disadvantage: roll twice, take the lower result
    UFUNCTION(BlueprintCallable, Category = "Dice")
    FDiceRollResult RollWithDisadvantage(int32 NumDice, int32 NumSides,
                                          int32 Modifier = 0);

    // Roll a stat check against a DC
    UFUNCTION(BlueprintCallable, Category = "Dice")
    bool StatCheck(int32 StatModifier, int32 DifficultyClass,
                   FDiceRollResult& OutRollResult);

    // Convenience functions
    UFUNCTION(BlueprintCallable, Category = "Dice")
    int32 RollD4();

    UFUNCTION(BlueprintCallable, Category = "Dice")
    int32 RollD6();

    UFUNCTION(BlueprintCallable, Category = "Dice")
    int32 RollD8();

    UFUNCTION(BlueprintCallable, Category = "Dice")
    int32 RollD10();

    UFUNCTION(BlueprintCallable, Category = "Dice")
    int32 RollD12();

    UFUNCTION(BlueprintCallable, Category = "Dice")
    int32 RollD20();

    // History
    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Dice")
    FDiceRollResult GetLastRoll() const;

    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Dice")
    TArray<FDiceRollResult> GetRollHistory() const;

    UFUNCTION(BlueprintCallable, Category = "Dice")
    void ClearHistory();

private:
    UPROPERTY()
    TArray<FDiceRollResult> RollHistory;

    FDiceRollResult PerformRoll(int32 NumDice, int32 NumSides, int32 Modifier);
    void RecordRoll(const FDiceRollResult& Result);
};
```

**DiceRollerSubsystem.cpp**
```cpp
#include "DiceRollerSubsystem.h"

FDiceRollResult UDiceRollerSubsystem::Roll(int32 NumDice, int32 NumSides,
                                            int32 Modifier)
{
    FDiceRollResult Result = PerformRoll(NumDice, NumSides, Modifier);
    RecordRoll(Result);
    return Result;
}

FDiceRollResult UDiceRollerSubsystem::RollWithAdvantage(int32 NumDice,
    int32 NumSides, int32 Modifier)
{
    FDiceRollResult Roll1 = PerformRoll(NumDice, NumSides, Modifier);
    FDiceRollResult Roll2 = PerformRoll(NumDice, NumSides, Modifier);
    FDiceRollResult Better = (Roll1.Total >= Roll2.Total) ? Roll1 : Roll2;
    RecordRoll(Better);
    return Better;
}

FDiceRollResult UDiceRollerSubsystem::RollWithDisadvantage(int32 NumDice,
    int32 NumSides, int32 Modifier)
{
    FDiceRollResult Roll1 = PerformRoll(NumDice, NumSides, Modifier);
    FDiceRollResult Roll2 = PerformRoll(NumDice, NumSides, Modifier);
    FDiceRollResult Worse = (Roll1.Total <= Roll2.Total) ? Roll1 : Roll2;
    RecordRoll(Worse);
    return Worse;
}

bool UDiceRollerSubsystem::StatCheck(int32 StatModifier,
    int32 DifficultyClass, FDiceRollResult& OutRollResult)
{
    OutRollResult = Roll(1, 20, StatModifier);
    return OutRollResult.Total >= DifficultyClass;
}

int32 UDiceRollerSubsystem::RollD4()  { return Roll(1, 4).Total; }
int32 UDiceRollerSubsystem::RollD6()  { return Roll(1, 6).Total; }
int32 UDiceRollerSubsystem::RollD8()  { return Roll(1, 8).Total; }
int32 UDiceRollerSubsystem::RollD10() { return Roll(1, 10).Total; }
int32 UDiceRollerSubsystem::RollD12() { return Roll(1, 12).Total; }
int32 UDiceRollerSubsystem::RollD20() { return Roll(1, 20).Total; }

FDiceRollResult UDiceRollerSubsystem::GetLastRoll() const
{
    if (RollHistory.Num() > 0)
    {
        return RollHistory.Last();
    }
    return FDiceRollResult();
}

TArray<FDiceRollResult> UDiceRollerSubsystem::GetRollHistory() const
{
    return RollHistory;
}

void UDiceRollerSubsystem::ClearHistory()
{
    RollHistory.Empty();
}

FDiceRollResult UDiceRollerSubsystem::PerformRoll(int32 NumDice,
    int32 NumSides, int32 Modifier)
{
    FDiceRollResult Result;
    Result.NumDice = NumDice;
    Result.NumSides = NumSides;
    Result.Modifier = Modifier;
    Result.Total = Modifier;

    for (int32 i = 0; i < NumDice; ++i)
    {
        int32 SingleRoll = FMath::RandRange(1, NumSides);
        Result.IndividualRolls.Add(SingleRoll);
        Result.Total += SingleRoll;
    }

    return Result;
}

void UDiceRollerSubsystem::RecordRoll(const FDiceRollResult& Result)
{
    RollHistory.Add(Result);
    if (RollHistory.Num() > 100)
    {
        RollHistory.RemoveAt(0);
    }
}
```

## The Three Magic Macros: UCLASS, UPROPERTY, UFUNCTION

These three macros are the bridge between C++ and Blueprints. Understanding them is the single most important thing in this module. You do not need to understand pointer arithmetic, template specialization, or the C++ standard library. You need to understand these three macros and when Claude uses each one.

### How the Macros Work (The 30-Second Version)

Unreal has a tool called the **Unreal Header Tool (UHT)** that runs before your code compiles. UHT scans your header files looking for these macros. When it finds them, it generates additional C++ code (called "reflection code") that tells the engine about your classes, properties, and functions. This generated code is what makes Blueprints, the Details panel, serialization, garbage collection, and networking all work.

Think of it like writing annotations on a building blueprint. The construction crew (the compiler) builds the structure. But the annotations (the macros) tell the city inspector (the engine) what each room is for, which doors are public entrances, and which walls are load-bearing.

### UCLASS(): Making Classes Visible to the Engine

Every class that interacts with the Unreal reflection system needs the `UCLASS()` macro. Without it, the engine does not know the class exists.

```cpp
UCLASS(Blueprintable, BlueprintType)
class TABLETOPQUEST_API UTQAbilityBase : public UObject
{
    GENERATED_BODY()

public:
    // Class content goes here
};
```

Let's break down every piece:

- `UCLASS(...)`: The macro itself. The specifiers inside the parentheses control how the class behaves.
- `TABLETOPQUEST_API`: An export macro that makes the class visible to other modules. It matches your module name in all caps with `_API` appended.
- `GENERATED_BODY()`: Must be the first line inside the class. UHT replaces this with generated reflection code. Forget it and compilation fails.

The specifiers you will see Claude use most often:

| Specifier | What It Does | When to Use It |
|-----------|-------------|----------------|
| `Blueprintable` | Allows creating Blueprint subclasses of this C++ class | When you want designers to extend the class in Blueprints |
| `BlueprintType` | This class can be used as a variable type in Blueprints | When Blueprints need to hold references to instances |
| `Abstract` | Cannot be instantiated directly, only subclassed | For base classes like "AbilityBase" that need concrete implementations |
| `NotBlueprintable` | Prevents Blueprint subclassing | For internal engine classes that should stay in C++ |
| `MinimalAPI` | Reduces the exported API for faster compilation | When the class is used internally but rarely referenced by other modules |

**For Tabletop Quest**: Most of your game system classes (combat manager, ability definitions, enemy data) will be `Blueprintable, BlueprintType` so you can create Blueprint variations. Your Warrior's Power Strike ability will be a Blueprint that extends your C++ ability base class.

### UPROPERTY(): Exposing Variables

This macro marks a variable for the reflection system. Without it, the engine has no idea the variable exists. The variable will not show up in the Details panel. It will not be saved when you save the level. It will not replicate in multiplayer. It will not be accessible from Blueprints.

```cpp
// Editable in the editor, readable in Blueprints
UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Combat Stats")
float BaseDamage = 10.0f;

// Fully accessible from Blueprints, but not editable in the editor Details panel
UPROPERTY(BlueprintReadWrite, Category = "Combat Stats")
float CurrentHealth;

// Editable everywhere, fully accessible, with a display clamp
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Combat Stats",
          meta = (ClampMin = "0", ClampMax = "30"))
int32 ArmorClass = 10;

// Visible in the editor but not editable (useful for debugging/monitoring)
UPROPERTY(VisibleAnywhere, BlueprintReadOnly, Category = "Runtime")
int32 CurrentTurnNumber = 0;
```

The most common UPROPERTY specifiers, organized by what they control:

**Editor visibility:**

| Specifier | Details Panel | Class Defaults | When to Use |
|-----------|--------------|----------------|-------------|
| `EditAnywhere` | Editable | Editable | Most common. Stats, configuration values. |
| `EditDefaultsOnly` | Read-only | Editable | Class-wide settings that should not change per instance. |
| `EditInstanceOnly` | Editable | Read-only | Per-instance overrides. |
| `VisibleAnywhere` | Read-only | Read-only | Display-only values. Current health, turn count. |

**Blueprint access:**

| Specifier | Can Read? | Can Write? | When to Use |
|-----------|-----------|------------|-------------|
| `BlueprintReadOnly` | Yes | No | Stats that Blueprints display but should not modify directly. |
| `BlueprintReadWrite` | Yes | Yes | Values Blueprints need to both read and change. |
| (neither) | No | No | Internal C++ variables that Blueprints should not touch. |

**Organization:**

| Specifier | What It Does |
|-----------|-------------|
| `Category = "Name"` | Groups properties under collapsible headers in the Details panel |
| `meta = (ClampMin = "0")` | Adds a minimum value slider in the editor |
| `meta = (ClampMax = "100")` | Adds a maximum value slider in the editor |
| `meta = (DisplayName = "Hit Points")` | Shows a different name in the editor |

**For Tabletop Quest**: Your character stats (Might, Finesse, Mind, Presence) will be `EditAnywhere, BlueprintReadWrite` so you can tweak base values in the editor, and Blueprints can modify them at runtime during level-ups and buff/debuff effects.

### UFUNCTION(): Making Functions Blueprint-Callable

This is where the real power lives. UFUNCTION marks a C++ function so Blueprints can call it. Without this macro, the function exists only in C++. With it, the function becomes a node you can drag into any Blueprint graph.

```cpp
// Blueprint can call this function. It appears as a node with an execution pin.
UFUNCTION(BlueprintCallable, Category = "Combat")
int32 CalculateDamage(int32 BaseDamage, int32 AttackerMight, int32 DefenderArmor);

// Pure function: no side effects, no execution pin. Used in expressions.
UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Combat")
bool IsAlive() const;

// C++ provides a default implementation. Blueprints can override it.
UFUNCTION(BlueprintNativeEvent, Category = "Combat")
void OnDamageTaken(float DamageAmount, AActor* DamageSource);

// No C++ implementation at all. Blueprint MUST provide one.
UFUNCTION(BlueprintImplementableEvent, Category = "Combat")
void OnDeath();
```

The critical UFUNCTION specifiers:

| Specifier | What It Does | When to Use |
|-----------|-------------|-------------|
| `BlueprintCallable` | Appears as a callable node in Blueprint graphs | 90% of your functions. This is the default choice. |
| `BlueprintPure` | No execution pin, used as an expression (like math) | Getters, calculators, checks. Anything that does not change state. |
| `BlueprintNativeEvent` | C++ provides a default, Blueprint can override | When you want C++ to handle the common case but allow per-Blueprint customization. |
| `BlueprintImplementableEvent` | No C++ implementation, Blueprint must provide one | When the behavior is entirely up to the Blueprint designer. |
| `Category = "Name"` | Organizes functions in Blueprint right-click search | Always include this. It makes functions easier to find. |

**The two you will use 90% of the time**: `BlueprintCallable` for functions Claude writes that you call from Blueprints, and `BlueprintNativeEvent` for functions where you want a sensible C++ default but the option to override behavior in specific Blueprints.

## USTRUCT: Custom Data Types for Blueprints

When you need to pass a bundle of related data around, you use USTRUCT. A dice roll result, a combat damage packet, a character stat block: these are all structs.

```cpp
USTRUCT(BlueprintType)
struct FCharacterStats
{
    GENERATED_BODY()

    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Stats")
    int32 Might = 10;

    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Stats")
    int32 Finesse = 10;

    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Stats")
    int32 Mind = 10;

    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Stats")
    int32 Presence = 10;

    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Derived")
    float MaxHealth = 100.0f;

    UPROPERTY(BlueprintReadWrite, Category = "Runtime")
    float CurrentHealth = 100.0f;

    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Derived")
    float MaxMana = 50.0f;

    UPROPERTY(BlueprintReadWrite, Category = "Runtime")
    float CurrentMana = 50.0f;
};
```

In Blueprints, `FCharacterStats` shows up as a variable type. You can:
- Create variables of this type
- **Break** the struct to access individual fields (Break node)
- **Make** a struct by setting all fields at once (Make node)
- Pass it to and from functions

Think of a struct like a character sheet: a single object that holds all the stats for one character. You pass the whole sheet around rather than passing six separate numbers.

### When to Use Structs vs. Classes

- **Structs** are value types. They get copied when passed around. Use them for data bundles: stat blocks, roll results, damage packets, loot entries. They are lightweight and straightforward.
- **Classes** (UObject subclasses) are reference types. They live on the heap and the engine manages their memory through garbage collection. Use them for systems with behavior and identity: ability definitions, AI controllers, quest trackers, the combat manager.

For Tabletop Quest: stat blocks, dice results, and damage packets are structs. The combat system, ability system, and AI DM interface are classes.

## UENUM: Blueprint-Friendly Enumerations

Enums define a set of named options. Tabletop Quest has combat modes, damage types, character classes, status effects, and equipment slots. All perfect candidates for enums.

```cpp
UENUM(BlueprintType)
enum class ECombatMode : uint8
{
    TurnBased      UMETA(DisplayName = "Turn-Based"),
    RealTime       UMETA(DisplayName = "Real-Time"),
    Transitioning  UMETA(DisplayName = "Transitioning")
};

UENUM(BlueprintType)
enum class EDamageType : uint8
{
    Physical   UMETA(DisplayName = "Physical"),
    Fire       UMETA(DisplayName = "Fire"),
    Ice        UMETA(DisplayName = "Ice"),
    Lightning  UMETA(DisplayName = "Lightning"),
    Poison     UMETA(DisplayName = "Poison"),
    Holy       UMETA(DisplayName = "Holy"),
    Necrotic   UMETA(DisplayName = "Necrotic"),
    Psychic    UMETA(DisplayName = "Psychic")
};

UENUM(BlueprintType)
enum class ECharacterClass : uint8
{
    Warrior  UMETA(DisplayName = "Warrior"),
    Rogue    UMETA(DisplayName = "Rogue"),
    Mage     UMETA(DisplayName = "Mage"),
    Cleric   UMETA(DisplayName = "Cleric"),
    Ranger   UMETA(DisplayName = "Ranger"),
    Bard     UMETA(DisplayName = "Bard")
};

UENUM(BlueprintType)
enum class EStatusEffect : uint8
{
    None       UMETA(DisplayName = "None"),
    Stunned    UMETA(DisplayName = "Stunned"),
    Poisoned   UMETA(DisplayName = "Poisoned"),
    Burning    UMETA(DisplayName = "Burning"),
    Frozen     UMETA(DisplayName = "Frozen"),
    Frightened UMETA(DisplayName = "Frightened"),
    Blessed    UMETA(DisplayName = "Blessed"),
    Hasted     UMETA(DisplayName = "Hasted")
};
```

In Blueprints, these appear as dropdown menus. Instead of remembering that damage type "2" means "Ice," you select "Ice" from a readable list. The `UMETA(DisplayName = ...)` controls what text appears in those dropdowns.

Why not just use integers? Because three months from now you will look at a Blueprint node that says "DamageType = 5" and have no idea what that means. With enums, it says "DamageType = Holy" and you know immediately.

## Building a Real Plugin: The Stat Calculator

Let's build a more substantial plugin. The **TabletopStats** plugin handles all the math behind character stats, ability modifiers, damage calculation, and XP progression. This is the kind of system that benefits from C++ because it gets called hundreds of times during combat resolution.

### The Prompt to Give Claude

> "Write a UE5 C++ class called UStatCalculatorSubsystem (UGameInstanceSubsystem) for a DnD-inspired RPG called Tabletop Quest. Blueprint-callable functions needed:
> - GetAbilityModifier: calculates modifier from ability score using DnD formula ((score - 10) / 2, rounded down)
> - CalculateArmorClass: from base armor value, Finesse modifier, and optional shield bonus
> - DoesAttackHit: compares attack roll + modifier against target AC, returns bool
> - CalculateDamage: takes base damage, damage type enum, arrays of resistances and vulnerabilities, and a critical hit flag. Returns a struct with raw damage, final damage, multiplier, and flags for resisted/vulnerable/critical
> - GetXPThreshold: XP needed for a given level (1-20)
> - GetLevelForXP: what level a given XP total corresponds to
> The damage type enum should be: Physical, Fire, Ice, Lightning, Poison, Holy, Necrotic, Psychic.
> Return calculation results through a USTRUCT with full breakdown."

### What Claude Generates

Here is a condensed version showing the key patterns:

```cpp
// StatCalculatorSubsystem.h
#pragma once

#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "StatCalculatorSubsystem.generated.h"

USTRUCT(BlueprintType)
struct FDamageCalcResult
{
    GENERATED_BODY()

    UPROPERTY(BlueprintReadOnly, Category = "Damage")
    float RawDamage = 0.0f;

    UPROPERTY(BlueprintReadOnly, Category = "Damage")
    float FinalDamage = 0.0f;

    UPROPERTY(BlueprintReadOnly, Category = "Damage")
    float DamageMultiplier = 1.0f;

    UPROPERTY(BlueprintReadOnly, Category = "Damage")
    bool bWasResisted = false;

    UPROPERTY(BlueprintReadOnly, Category = "Damage")
    bool bWasVulnerable = false;

    UPROPERTY(BlueprintReadOnly, Category = "Damage")
    bool bWasCritical = false;
};

UCLASS()
class TABLETOPSTATS_API UStatCalculatorSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Stats")
    int32 GetAbilityModifier(int32 AbilityScore) const;

    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Stats")
    int32 CalculateArmorClass(int32 BaseArmor, int32 FinesseModifier,
                               int32 ShieldBonus = 0) const;

    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Combat")
    bool DoesAttackHit(int32 AttackRoll, int32 AttackModifier,
                       int32 TargetAC) const;

    UFUNCTION(BlueprintCallable, Category = "Combat")
    FDamageCalcResult CalculateDamage(float BaseDamage,
        EDamageType DamageType,
        const TArray<EDamageType>& Resistances,
        const TArray<EDamageType>& Vulnerabilities,
        bool bIsCritical = false);

    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Progression")
    int32 GetXPThreshold(int32 Level) const;

    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Progression")
    int32 GetLevelForXP(int32 TotalXP) const;
};
```

The implementation follows the DnD formulas you specified. `GetAbilityModifier` uses `FMath::FloorToInt((AbilityScore - 10) / 2.0f)`. `CalculateDamage` checks if the damage type appears in the resistances array (halve damage) or vulnerabilities array (double damage), and doubles again on critical. Clean, fast, tested math that you never need to write yourself.

### Wiring It Into Tabletop Quest

Once compiled, here is how a Warrior's Power Strike flows through your Blueprints:

1. Player activates Power Strike ability
2. Blueprint calls `Roll(1, 20, AttackModifier)` on the DiceRoller subsystem
3. Blueprint calls `DoesAttackHit(RollResult.Total, MightModifier, EnemyAC)` on the StatCalculator
4. If the attack hits, Blueprint calls `Roll(1, 10, MightModifier)` for weapon damage
5. Blueprint calls `CalculateDamage(DamageRoll, Physical, EnemyResistances, EnemyVulnerabilities, bWasCrit)` on the StatCalculator
6. Blueprint applies `FinalDamage` to the enemy's health component
7. Blueprint checks `bWasResisted` and `bWasVulnerable` to display the correct combat text ("Resisted!", "Vulnerable!", "Critical Hit!")

The Blueprint graph reads like a flowchart. The heavy math lives in C++. Best of both worlds.

## Game Instance Subsystems: The Perfect Home for Game Systems

Both the DiceRoller and StatCalculator use `UGameInstanceSubsystem`. This is an important architectural decision for Tabletop Quest, and you should understand why.

### What is a Subsystem?

A subsystem is a class that the engine automatically creates and manages. You never call `new` or worry about lifecycle. The engine creates exactly one instance at the right time and destroys it when appropriate.

Think of subsystems like built-in staff at a hotel. The hotel (the engine) hires a concierge (your subsystem) when it opens, and the concierge is always available at the front desk. You do not need to find the concierge, create the concierge, or worry about when the concierge goes home. The hotel handles all of that.

There are several subsystem types:

| Subsystem Type | Lifetime | Good For |
|---------------|----------|----------|
| `UGameInstanceSubsystem` | Entire game session (survives level changes) | Core game systems |
| `UWorldSubsystem` | Current world/level only | Level-specific logic |
| `ULocalPlayerSubsystem` | Per local player | Player-specific settings |
| `UEngineSubsystem` | Entire engine session | Global utilities |

### Why GameInstanceSubsystem for Tabletop Quest?

Your combat system, stat calculator, dice roller, and quest tracker all need to survive level transitions. When the player zooms from the tabletop view into a dungeon (loading a new sublevel), the game world changes, but these systems must persist. `UGameInstanceSubsystem` lives as long as the game is running, regardless of which level is loaded.

This is particularly important for the tabletop zoom mechanic. The player is looking at the tabletop (Level A), zooms into a dungeon (Level B), fights some goblins, zooms back out to the tabletop (Level A again). Your combat tracker, dice history, and quest state cannot be destroyed and recreated every time a level loads. `UGameInstanceSubsystem` solves this automatically.

### Accessing Subsystems from Blueprints

In any Blueprint:

1. Right-click in the event graph
2. Search for the subsystem name (e.g., "Dice Roller Subsystem")
3. Use the "Get Subsystem" node that appears
4. Call functions on it directly

No spawning, no finding, no storing references. The engine guarantees exactly one instance exists and gives it to you on demand. It is the cleanest access pattern in all of UE5.

## Delegates and Events: C++ Talking to Blueprints

Sometimes your C++ code needs to tell Blueprints that something happened. "The combat mode switched." "A new round started." "A character leveled up." "A critical hit was scored." This is where delegates come in.

Think of delegates like a mailing list. Your C++ code says "anyone who cares about combat mode changes, I will notify you when it happens." Blueprints subscribe to the list. When the event fires, every subscriber gets the message.

### Dynamic Multicast Delegates

These are the most Blueprint-friendly delegate type. You declare them in C++ and Blueprints bind to them using the "Assign" node.

```cpp
// Declare the delegate signature (goes outside the class, usually in the header)
DECLARE_DYNAMIC_MULTICAST_DELEGATE_TwoParams(
    FOnCombatModeChanged,
    ECombatMode, OldMode,
    ECombatMode, NewMode
);

DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(
    FOnNewRoundStarted,
    int32, RoundNumber
);

DECLARE_DYNAMIC_MULTICAST_DELEGATE_ThreeParams(
    FOnDamageDealt,
    AActor*, Attacker,
    AActor*, Defender,
    float, DamageAmount
);
```

Inside the class, you expose them as BlueprintAssignable properties:

```cpp
UCLASS()
class UCombatManagerSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    UPROPERTY(BlueprintAssignable, Category = "Combat Events")
    FOnCombatModeChanged OnCombatModeChanged;

    UPROPERTY(BlueprintAssignable, Category = "Combat Events")
    FOnNewRoundStarted OnNewRoundStarted;

    UPROPERTY(BlueprintAssignable, Category = "Combat Events")
    FOnDamageDealt OnDamageDealt;

    UFUNCTION(BlueprintCallable, Category = "Combat")
    void SwitchCombatMode(ECombatMode NewMode);

private:
    ECombatMode CurrentMode = ECombatMode::TurnBased;
};
```

When the combat mode switches, the C++ code broadcasts to all listeners:

```cpp
void UCombatManagerSubsystem::SwitchCombatMode(ECombatMode NewMode)
{
    ECombatMode OldMode = CurrentMode;
    CurrentMode = NewMode;
    OnCombatModeChanged.Broadcast(OldMode, NewMode);
}
```

In Blueprints, you drag off `OnCombatModeChanged`, choose "Assign," and wire up whatever should happen. Your UI updates its layout. The camera shifts perspective. Enemy AI switches behavior patterns. Ability cooldown displays change format. All triggered from one C++ broadcast.

### Why This Matters for Tabletop Quest

The dual combat system (turn-based and real-time, switchable mid-fight) is a core pillar of the game. When the player switches modes, dozens of things need to react simultaneously:

- UI toggles between action point menus and cooldown bars
- Camera shifts between overhead tactical view and third-person action view
- Enemy AI switches between initiative-based planning and real-time behavior trees
- Ability costs convert between "action points" and "cooldown timers"
- Grid overlay appears or disappears
- Turn order tracker shows or hides

A single delegate broadcast from C++ triggers all of these responses in Blueprints. Each Blueprint subscribes to the events it cares about and handles its own response. No Blueprint needs to know about any other Blueprint. Clean, modular, maintainable.

## Data Assets: Configuration Without Recompilation

Here is a pattern Claude will use constantly for Tabletop Quest: Data Assets. These are editor-side asset files that hold configuration data. Instead of hardcoding the Goblin's stats in C++ (which requires recompilation to change), you define the data structure in C++ and then create asset files in the editor that hold the actual values.

```cpp
UCLASS(BlueprintType)

{
    GENERATED_BODY()

public:
    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Identity")
    FString EnemyName;

    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Identity")
    ECharacterClass EnemyClass;

    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Stats")
    FCharacterStats BaseStats;

    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Combat")
    int32 ChallengeRating = 1;

    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Combat")
    TArray<EDamageType> Resistances;

    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Combat")
    TArray<EDamageType> Vulnerabilities;

    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Combat")
    int32 XPReward = 50;

    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Visuals")
    TSoftObjectPtr<USkeletalMesh> Mesh;

    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Visuals")
    TSoftObjectPtr<UAnimBlueprint> AnimBlueprint;
};
```

Now in the Content Browser, you right-click, choose **Miscellaneous > Data Asset**, and select `EnemyDataAsset`. You get a form where you fill in:

- **Goblin**: 8 HP, Finesse 14, Challenge Rating 1, Vulnerable to Fire, 25 XP
- **Skeleton Warrior**: 13 HP, Might 12, Challenge Rating 2, Resistant to Physical, Vulnerable to Holy, 50 XP
- **Shadow Wraith**: 22 HP, Mind 16, Challenge Rating 4, Resistant to Physical/Ice, Vulnerable to Holy/Fire, 200 XP
- **Ancient Dragon**: 200 HP, all stats 20+, Challenge Rating 15, Resistant to Fire/Lightning, 5000 XP

No recompilation needed. You tweak the Goblin's HP from 8 to 10, save the asset, and the change takes effect immediately. Your combat system just loads the data asset for whichever enemy type it spawns.

For the entire Tabletop Quest bestiary (every creature in the enemy bestiary document), Data Assets are the perfect solution. Claude writes the C++ definition once, and you create as many enemy types as you want purely in the editor.

## Common Compilation Errors (and How to Fix Them)

When pasting Claude-generated code, you will occasionally hit compilation errors. Here are the most common ones, what they mean, and how to resolve them.

### "Missing GENERATED_BODY()"

Every UCLASS and USTRUCT needs `GENERATED_BODY()` as the first line inside the class/struct body. If it is missing, UHT cannot generate the reflection code and compilation fails with a cryptic error.

**Fix**: Add `GENERATED_BODY()` right after the opening brace of the class or struct.

### "Unrecognized type 'ClassName'"

A header file is referencing a type but has not included the header that defines it.

**Fix**: Find the class in the UE5 API docs and add the correct `#include`. Or tell Claude the error and Claude will add the missing include.

### "Cannot find module 'ModuleName'"

Your `Build.cs` is missing a dependency module. If Claude's code uses `UGameplayAbility`, you need `"GameplayAbilities"` in your dependency list.

**Fix**: Open your `.Build.cs` file and add the missing module name to `PublicDependencyModuleNames`.

Common module dependencies:

| If You Use | Add This Module |
|-----------|----------------|
| `UGameplayAbility`, `UAbilitySystemComponent` | `"GameplayAbilities"` |
| `FGameplayTag` | `"GameplayTags"` |
| `UNavigationSystemV1` | `"NavigationSystem"` |
| `UWidgetComponent` | `"UMG"` |
| `UInputAction` | `"EnhancedInput"` |
| `UAIController` | `"AIModule"` |

### "'_API' macro mismatch"

The API export macro (like `TABLETTOPDICE_API`) must match your module name exactly, in all caps. If your module is `TabletopDice`, the macro should be `TABLETTOPDICE_API`.

**Fix**: Check the module header file for the correct macro name and use it consistently.

### The Nuclear Option: Clean Rebuild

If you get truly stuck with mysterious compilation errors:

1. Close the UE5 editor completely
2. Delete the `Binaries/`, `Intermediate/`, and `DerivedDataCache/` folders from your project
3. Reopen the project from the .uproject file
4. UE5 will do a full rebuild from scratch

This takes longer but eliminates stale cached data that can cause phantom errors.

### Hot Reload vs. Live Coding vs. Full Restart

UE5 offers multiple ways to reload compiled code:

- **Hot Reload**: The legacy method. Unreliable for anything beyond trivial changes. Can cause crashes, stale data, and mysterious behavior. Avoid it.
- **Live Coding** (Ctrl+Alt+F11): The modern approach. More reliable than Hot Reload, especially for function body changes. Still struggles with header changes (new properties, new functions, changed signatures).
- **Full Restart**: Close the editor, compile from your IDE, reopen. Takes 30-60 seconds but is 100% reliable. Always works.

For Tabletop Quest, use Live Coding when you are making small changes to function implementations. Use a full restart when you add new classes, change class hierarchies, or add/remove UPROPERTY/UFUNCTION macros.

## Testing Your Plugins

Every plugin you create should be testable in isolation. Here is a lightweight approach that does not require a testing framework.

### Blueprint Test Actor

Create a Blueprint Actor called `BP_PluginTester`. In its BeginPlay event:

1. Get the subsystem you want to test
2. Call its functions with known inputs
3. Print the results to screen using `Print String`
4. Compare against expected values

Drop this actor in a test level, hit Play, and watch the output. Quick, repeatable, catches regressions.

### Example: Testing the Dice Roller

```
BeginPlay
  |
  Get DiceRollerSubsystem
  |
  Roll(2, 6, 0) --> Print "2d6 = {Total} (expected: 2-12)"
  |
  Roll(1, 20, 5) --> Print "d20+5 = {Total} (expected: 6-25)"
  |
  RollWithAdvantage(1, 20, 0) --> Print "Advantage d20 = {Total}"
  |
  RollWithDisadvantage(1, 20, 0) --> Print "Disadvantage d20 = {Total}"
  |
  StatCheck(3, 15, OutResult) --> Print "DC15 check: {Pass/Fail}, rolled {Total}"
```

### Automated Testing with Python

Claude can also write Python scripts that validate your plugins through the editor's Python API:

```python
import unreal
```
