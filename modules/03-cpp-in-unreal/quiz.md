## Answer Key

### Answer 1: B

`GENERATED_BODY()` is a placeholder that the Unreal Header Tool (UHT) replaces with generated reflection and serialization code. This code lets the engine know about the class's properties, functions, and inheritance at runtime. Without it, UCLASS and USTRUCT declarations will fail to compile.

### Answer 2

Claude must include `UFUNCTION(BlueprintCallable, Category = "SomeCategory")`. Once compiled, the function appears as a draggable node in the Blueprint editor's right-click context menu. You search for it by name and wire it into any event graph.

### Answer 3: B

`UGameInstanceSubsystem` persists for the entire game session, surviving level loads and transitions. This is critical for Tabletop Quest because the player frequently zooms between the tabletop view (one level) and dungeon/overworld locations (other levels). `UWorldSubsystem` would be destroyed and recreated on every level change, losing all combat state and dice history.

### Answer 4: C

`EditAnywhere` makes the property editable in both the Details panel and class defaults. `BlueprintReadOnly` lets Blueprints read the value but not write it. This is ideal for configuration values like base stats that you set in the editor but should not be accidentally overwritten by gameplay code at runtime.

### Answer 5

Edit the plugin's `.Build.cs` file. Add `"GameplayAbilities"` to the `PublicDependencyModuleNames` array:

```csharp
PublicDependencyModuleNames.AddRange(new string[]
{
    "Core", "CoreUObject", "Engine", "GameplayAbilities"
});
```

### Answer 6: False

The entire point of the Claude C++ workflow is that Claude handles the C++ complexity. You need to understand what the code does (at a high level), where to put the generated files (.h in Public/, .cpp in Private/), and how to use the resulting Blueprint nodes. You do not need to understand pointers, memory management, or template metaprogramming.

### Answer 7: B

`BlueprintCallable` creates a node with execution pins (the white arrows that control flow). The function runs when execution reaches it. `BlueprintPure` creates a node without execution pins, used for calculations and queries with no side effects (like `IsAlive()` or `GetAbilityModifier()`). Pure functions plug directly into other nodes' input pins.

### Answer 8

Use a **Dynamic Multicast Delegate**. Declare it with `DECLARE_DYNAMIC_MULTICAST_DELEGATE` (with appropriate parameter macros). Expose it as a UPROPERTY with the `BlueprintAssignable` specifier. In Blueprints, drag off the property, choose "Assign," and bind event handlers. When the C++ code calls `.Broadcast()`, all bound Blueprint handlers fire simultaneously.

### Answer 9: B

A Data Asset is an asset file created in the editor that holds configuration data defined by a C++ class (inheriting from `UPrimaryDataAsset`). For the bestiary, you define the data structure once in C++ (enemy name, stats, resistances, vulnerabilities, XP reward, mesh reference) and then create individual Data Assets per enemy in the editor. Changing a Goblin's HP from 8 to 10 requires editing the asset and saving. No recompilation needed.

### Answer 10

The complete workflow:

1. **Identify the need**: Decide what system Tabletop Quest needs next.
2. **Write a detailed prompt for Claude**: Specify class names, base classes, functions with parameters and return types, structs, enums, and which UE5 macros to use.
3. **Review Claude's output**: Check for UCLASS/UFUNCTION/UPROPERTY macros, GENERATED_BODY(), correct includes, and Build.cs dependencies.
4. **Paste into the plugin**: .h files in `Public/`, .cpp files in `Private/`.
5. **Compile**: Click the Compile button. Send any errors to Claude for fixes.
6. **Wire up in Blueprints**: Right-click in a Blueprint graph, search for function names, drag and connect.
7. **Test**: Use a Blueprint test actor or Python script to verify behavior.
8. **Iterate**: Describe issues to Claude, get updated code, paste, compile, test again.
