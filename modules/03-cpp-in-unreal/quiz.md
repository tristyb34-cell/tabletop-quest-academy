## Answer Key

**Question 1**: B) It expands to reflection and serialization code generated by the Unreal Header Tool. UHT scans for UCLASS and GENERATED_BODY, then generates code that integrates the class with Unreal's reflection system, enabling features like Blueprint exposure, serialization, and garbage collection.

**Question 2**: UPROPERTY tells the garbage collector that your class holds a reference to that UObject. Without UPROPERTY, the garbage collector does not know about the pointer. It might destroy the object, leaving you with a dangling pointer (a pointer to freed memory). Accessing a dangling pointer causes undefined behavior, typically a crash. Rule: every UObject pointer stored as a class member should be a UPROPERTY.

**Question 3**: C) EditAnywhere. This makes the property editable both on placed instances in the level and on the class defaults (the Blueprint editor's defaults tab). EditInstanceOnly restricts editing to placed instances only. EditDefaultsOnly restricts editing to the class defaults only. VisibleAnywhere makes it visible but not editable.

**Question 4**: `BlueprintCallable` exposes a function to Blueprints with an execution pin (the white wire). It represents an action that does something. Example: `ApplyDamage(float Amount)`, a function that modifies health. `BlueprintPure` exposes a function to Blueprints without an execution pin. It represents a query that returns data without side effects. Example: `GetHealthPercent()`, a function that calculates and returns a value without changing any state. Pure functions can be connected directly to data input pins.

**Question 5**: C) BlueprintNativeEvent. This creates a function with a default C++ implementation (in a method suffixed with `_Implementation`) that Blueprint subclasses can override. `BlueprintImplementableEvent` has no C++ implementation at all; it is declared in C++ but defined entirely in Blueprints.

**Question 6**: The header file (.h) declares the interface of a class: what properties it has, what functions it offers, and what types it uses. The implementation file (.cpp) contains the actual logic of those functions. C++ separates them for compilation efficiency: when another file needs to reference your class, it only needs to read the header (the "what"), not the full implementation (the "how"). This reduces compile times in large projects and creates a clean separation between contract and behavior. Think of the header as a restaurant menu (what is available) and the .cpp as the kitchen (how it is made).

**Question 7**: C) TMap. A TMap is a key-value dictionary that allows you to store and retrieve values using a key. `TMap<FString, int32>` would let you look up integer values by string keys. TArray is an ordered list (access by index). TSet is an unordered collection of unique values (no key-value pairs).

**Question 8**: In this pattern, you write core systems, base classes, and performance-critical logic in C++. You then expose properties (UPROPERTY) and functions (UFUNCTION) to Blueprints. Designers and gameplay programmers create Blueprint subclasses that set specific values, add visual components, and wire up high-level behavior. This is best practice because: (1) C++ provides performance and clean code organization for foundational systems, (2) Blueprints provide fast iteration, since designers can tweak values and test without recompiling, (3) the C++ base ensures consistency and correctness while Blueprints provide flexibility, (4) it separates engineering concerns from design concerns, which scales well on teams.

**Question 9**: C) UActorComponent. An ActorComponent is a component that holds data and logic but has no transform (no position, rotation, or scale in the world). It is perfect for systems like health, inventory, or stats that need to be attached to actors but do not need spatial representation. USceneComponent would work too, but it carries unnecessary transform overhead. AActor and APawn are standalone entities, not components.

**Question 10**: A delegate is a type-safe function pointer that allows one object to call a function on another object without a direct compile-time dependency. A **single-cast delegate** can bind to exactly one function. It is a one-to-one callback. Use it when only one listener needs to respond. A **multi-cast delegate** can bind to multiple functions. When broadcast, it calls all bound functions. It is one-to-many communication. Use it when multiple systems need to react to the same event. A **dynamic multi-cast delegate** is a multi-cast delegate that is compatible with Unreal's reflection system, meaning Blueprints can bind to it. You declare it with `DECLARE_DYNAMIC_MULTICAST_DELEGATE` and mark the property with `BlueprintAssignable`. Choose the dynamic version when you need Blueprints to subscribe to C++ events, which is the standard approach for the C++/Blueprint cooperation pattern.
