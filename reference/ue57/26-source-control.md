## Source Control

### Perforce

#### Setup

1. Install Perforce Helix Core server and client (p4v or p4).
2. In UE Editor: **Edit > Editor Preferences > Source Control > Provider**: select **Perforce**.
3. Configure: Server (e.g., `ssl:perforce.yourcompany.com:1666`), User Name, Workspace.
4. Click **Accept Settings**.

#### Workspace Configuration

- Workspace view should map the UE project root.
- Recommended typemap additions for `.uasset` and `.umap` files (binary, exclusive checkout):

```
binary+l //depot/.../*.uasset
binary+l //depot/.../*.umap
binary+w //depot/.../*.ini
text //depot/.../*.cpp
text //depot/.../*.h
```

#### Checkout and Submit

- Files are checked out automatically when modified in the editor (if auto-checkout is enabled).
- Manual checkout: Right-click asset in Content Browser > **Source Control > Check Out**.
- Submit: **File > Submit to Source Control** or right-click > **Source Control > Check In**.
- Revert: **Source Control > Revert**.
- Changelist management available in **Source Control > View Changelists**.

> **In your games:**
> - Perforce is designed for large teams. Since you are a solo developer, **Git with LFS is the better choice** for both projects. Skip the Perforce section unless you join a studio later. The key takeaway: UE binary assets (`.uasset`, `.umap`) need special handling in any source control system, which is why Git LFS exists.

### Git with Git LFS

#### Setup

1. Initialize a Git repo in the project root: `git init`.
2. Install Git LFS: `git lfs install`.
3. In UE Editor: **Edit > Editor Preferences > Source Control > Provider**: select **Git** (or **Git LFS 2** if available via plugin).
4. Configure the Git binary path if not in system PATH.

#### .gitignore for UE5

```gitignore
# Build artifacts
Binaries/
Intermediate/
DerivedDataCache/
Build/

# Saved folder
Saved/

# IDE files
.vs/
.vscode/
*.sln
*.suo
*.opensdf
*.sdf
*.VC.db
*.VC.opendb

# OS files
.DS_Store
Thumbs.db

# Compiled shaders
*.shadertext
*.ushaderbytecode
```

#### LFS Tracking

```bash
# Track binary UE asset types with LFS
git lfs track "*.uasset"
git lfs track "*.umap"
git lfs track "*.png"
git lfs track "*.jpg"
git lfs track "*.bmp"
git lfs track "*.tga"
git lfs track "*.exr"
git lfs track "*.hdr"
git lfs track "*.wav"
git lfs track "*.mp3"
git lfs track "*.ogg"
git lfs track "*.mp4"
git lfs track "*.mov"
git lfs track "*.fbx"
git lfs track "*.obj"
git lfs track "*.abc"
```

Commit `.gitattributes` after setting up LFS tracking.

> **In your games:**
> - **DnD RPG**: This is your primary project. Set up Git LFS from the very start. Your repo will have:
>   - `.uasset` files: Blueprints for characters, enemies, abilities, inventory items, PCG graphs, Niagara systems
>   - `.umap` files: Dungeon maps, tabletop overview map, main menu
>   - `.fbx` files: Character meshes, enemy meshes, dungeon props, chess pieces (if shared)
>   - `.png`/`.tga` textures: Materials for dungeon walls, spell effects, UI elements
>   - `.wav`/`.ogg` audio: Sword clashes, spell sounds, ambient dungeon audio
>   - All of these are binary and must go through LFS. Without LFS, your repo would balloon in size.
> - **Wizard's Chess**: Same LFS setup, but the repo stays small. Fewer assets overall: board mesh, 6 unique piece types, particle textures, and a handful of materials.
> - **Both projects**: Your `.gitignore` above covers the essentials. The `Saved/`, `Intermediate/`, `DerivedDataCache/`, and `Binaries/` folders are regenerated by the engine, so never commit them. Your C++ source files (`.cpp`, `.h`) are regular text and do not need LFS.

### SVN Support

- **Edit > Editor Preferences > Source Control > Provider**: select **Subversion**.
- Configure repository URL and credentials.
- SVN support is less commonly used than Perforce or Git for UE projects.
- Works similarly to Perforce: checkout, modify, commit workflow.

> **In your games:**
> - Skip SVN. Git with LFS is the modern standard for solo UE developers. Your existing GitHub knowledge transfers directly.

### Built-in Revision Control

- **Edit > Editor Preferences > Source Control**: Configure the source control provider.
- Settings: Auto-checkout on asset modification, prompt for checkout on edit, submit content hotfixes.
- **Toolbar > Source Control** button shows connection status and provides quick access to submit/revert.
- Status icons on assets: green circle (up to date), red check (checked out by you), red circle (checked out by someone else), yellow question mark (not in source control), blue plus (marked for add).

> **In your games:**
> - **DnD RPG**: Enable the Git provider in Editor Preferences so you see status icons on assets in the Content Browser. The blue plus icon (marked for add) is especially useful when you create new Blueprints, Niagara systems, or PCG graphs and want to confirm they are tracked before committing.
> - **Wizard's Chess**: Same setup. The yellow question mark icon helps you catch untracked files, like a new material you created for a chess piece but forgot to `git add`.
> - **Commit strategy for both**: Commit after completing logical units of work. For the RPG, that might be "added goblin enemy Blueprint with AI behavior tree" or "created fireball ability with Niagara VFX." For chess, "added knight capture animation and particle effect."

### Diffing Assets

#### Blueprint Diff

- Right-click a Blueprint asset > **Source Control > Diff Against Depot** (Perforce) or **Diff** (Git).
- Opens a side-by-side Blueprint graph comparison showing added, removed, and modified nodes.
- Node-level diff highlighting: green (added), red (removed), yellow (modified).
- Variable and function list changes shown in the diff panel.

#### Material Diff

- Right-click a Material asset > **Diff Against Depot/Diff**.
- Shows side-by-side material graph comparison.
- Node connections, parameter values, and graph layout differences highlighted.

> **In your games:**
> - **DnD RPG**: Blueprint Diff is invaluable when you change a character's ability Blueprint and want to see exactly what nodes were added or modified. If a spell suddenly behaves differently after an edit, diffing against the previous version reveals the change instantly.
> - **Wizard's Chess**: Use Material Diff if you tweak the magical glow material on chess pieces and want to compare the old look vs the new parameter values.
> - **Limitation**: These visual diffs only work within the UE editor. For C++ changes, use standard `git diff` on the command line or in your IDE.
